# interaction-audit — designing a semantic chrome-vs-handler integrity agent

Teaching material for Claude Code. When you bootstrap a `.claude/` directory, this doc teaches you HOW to author an agent that catches a class of UX bug invisible to both code review and visual review: **what the chrome PROMISES vs. what the handler ACTUALLY does.**

## When to ship one (applicability gate)

Ship an interaction-audit agent when:

- The project has **multi-element interactive screens** — forms, dashboards, lists with actions, wizards with cards + CTAs.
- The user has experienced or anticipates "the user taps the button and nothing happens" / "the user can't tell which control does what" bugs.
- Visual polish reviews have been passing while the actual interactions feel off.

Skip when:

- The project has only trivial single-element interactions (single submit button at the bottom of a form).
- The project is purely view-only / read-only (no interactive elements to audit).
- The project's UI is wholly generated by a library where the developer doesn't author interaction handlers.

## Why it matters — what this catches that nothing else does

Three failure modes that survive both `code-review.md` and `ux-audit.md`:

1. **Dead chrome.** A button / pressable / card visually promises action — chevron, solid CTA color, hover state — but its handler never fires on the primary user path. The user taps; nothing happens (or worse, something happens that the user didn't expect). See `../examples/the-button-that-never-fired.md` for the canonical paradigm.

2. **Redundant affordances.** Two interactive elements lead to the same outcome. The user must choose between them and gets no feedback on the choice. The visual hierarchy lies: it suggests they're different actions when they're not.

3. **Optical-group disconnects.** A control block A modifies a target field B, but lives far from it on the screen — separated by unrelated content. The user can't see the relationship. Tapping A appears to do nothing because the feedback is elsewhere on the screen.

These bugs are invisible to:
- **Code review** — the diff is fine; the handler returns the right value; types match.
- **Visual review** — each element in isolation is well-styled; the chrome is consistent with the design system.
- **E2E tests** — tests assert that the primary flow completes; they don't ask "could the user have taken a different path?"

Only an audit that **traces each affordance's visible promise to its actual behavior** catches them. That's the interaction audit's job.

## Core methodology — the affordance-vs-behavior table

The agent's central output, per screen, is a table:

| # | Element (testID / label) | Chrome promises | Handler does | Match? | Other elements doing same? |
|---|---|---|---|---|---|

Each interactive element gets one row. Filling the table:

### Step 1 — Capture the surface

- Screenshot the screen.
- Inspect the view hierarchy.
- Identify every visually-interactive element (Pressable, Button, Tab, Link, Card-with-onPress, anything that visually invites a tap).

### Step 2 — For each element, fill the row

**Chrome promises** — what does the visual design tell the user this element does? Examples:
- "Tap to navigate" (chevron + list-row chrome)
- "Tap to commit current selection" (solid CTA in primary color)
- "Tap to toggle" (switch / radio chrome)
- "Tap to dismiss" (X icon top-right of modal)

**Handler does** — read the actual code. Grep the testID / label. Trace the onPress handler 1–2 hops. Don't infer from chrome — read the code.

**Match?** — three states:
- ✓ chrome promise matches behavior exactly
- ✗ mismatch (chevron promises navigation but handler is decorative; CTA promises commit but handler is empty)
- ⚠ ambiguous (works but unclear which action the user thinks they're taking)

**Other elements doing same?** — does any OTHER element on the screen achieve the same outcome? If yes, there's a redundant-affordance candidate.

### Step 3 — Test by tapping, not just reading

The agent should TAP the affordance and observe what actually happens. Reading the code is necessary but insufficient — runtime behavior can diverge from static code:
- The handler reads stale state (closure trap, see the war story).
- The handler depends on an effect that hasn't fired yet.
- The element is intercepted by an overlay (see `../examples/the-button-that-never-fired.md`).
- The handler runs but the effect is invisible (writes to a store the screen doesn't read).

The audit's affordance-vs-behavior column is INFORMED by tapping, not by reading alone.

### Step 4 — Pattern-detect across the table

After every row is filled, scan for the three failure-mode patterns:

- **Dead chrome**: rows with ✗ — the element looks tappable but doesn't behave as the chrome promises.
- **Redundant affordances**: rows where "Other elements doing same?" is non-empty — two paths to the same outcome.
- **Optical-group disconnects**: cross-reference each element's screen position with its target. If element A at position (100, 200) modifies field B at position (300, 600), with unrelated content in between, that's a disconnect.

### Step 5 — Produce a graded report

Per `quality-rubric.md`'s S/A/B/C/D/F scale, with patterns explicitly named:

```markdown
## Interaction Audit — <screen> — <date>

### Overall: <S/A/B/C/D/F>

### Affordance-vs-behavior table
<the full table>

### Dead chrome findings
<each: element, what chrome promises, what handler actually does, severity>

### Redundant affordance findings
<each: paired elements, shared outcome, recommended consolidation>

### Optical-group disconnect findings
<each: element + target field, screen distance / separation, recommended fix>

### Highest-ROI move
<one concrete action to resolve the most-severe finding>
```

## How to derive THIS project's specifics

Before authoring the agent, gather:

1. **The project's interactive element types.** React Native: Pressable / Touchable / Button. Web: button / a / div-with-onClick / form elements. iOS native: UIButton / UIControl + accessibility traits. The agent's enumeration grep needs the project's specific element-type strings.

2. **The project's handler-tracing pattern.** Where do interactive elements call into? `onPress` → handler function → mutation? `onClick` → router push? `formAction` → server action? The agent's "read the handler" step needs to know the project's pattern.

3. **The project's testID / element-identifier convention.** Some projects use `testID`, some `data-testid`, some `id`, some accessibility labels. The agent's grep needs the right attribute name.

4. **The capture / inspection commands.** Project-specific. View-hierarchy inspection: web → DOM accessibility tree via DevTools / `axe-core`; iOS sim → `maestro hierarchy`; native iOS → Xcode accessibility inspector.

5. **The project's interactive-redundancy history.** Has the user shipped a "two buttons doing the same thing" bug before? That history primes the agent for the patterns to look for.

## Authoring the agent

The final agent (typically `.claude/agents/interaction-audit.md`) should specify:

1. **When to invoke** — default: after any UI screen edit, BEFORE `ux-audit` runs (per `audit-routing.md` canonical order — semantic before visual, because semantic fixes shift layout).
2. **What this agent is NOT** — explicit refusal list: not visual polish, not arc / IA, not token discipline, not copy register.
3. **The capture commands** — project-specific.
4. **The audit procedure** — five steps above.
5. **The affordance-vs-behavior table format** — same shape as above.
6. **The three failure-mode patterns** — dead chrome / redundant affordance / optical-group disconnect.
7. **The "tap before grading" rule** — runtime behavior over static code reading.
8. **The graded report format** — S/A/B/C/D/F + per-pattern findings.

## Cross-references

- `ux-audit.md` — visual polish. Runs AFTER interaction-audit per canonical pipeline order; semantic fixes shift layout, so visual goes last.
- `a11y-audit.md` — accessibility. Runs in PARALLEL with interaction-audit — orthogonal dimensions.
- `audit-routing.md` — the canonical pipeline (token → semantic + a11y || → visual) is binding.
- `../examples/the-button-that-never-fired.md` — paradigm for dead chrome. The audit's "tap and observe" rule is the diagnostic this war story would have unlocked.
- `quality-rubric.md` — the rubric's "tone mismatch" pitfall and "hierarchy violation" pitfall map to interaction-audit findings.

## Anti-patterns in the agent you write

- **Reading the handler without tapping.** Static reading misses runtime divergence (stale closures, async races, intercepting overlays). The audit must include actual interaction.

- **Inferring chrome from code, not screen.** "Chrome promises" is a property of the rendered output, not the source. The agent reads the screenshot for chrome, the code for behavior.

- **Skipping the "other elements doing same" column.** That column is how redundant-affordance findings emerge. Without it, the audit catches dead chrome but not redundancy.

- **Grading per-element instead of pattern-detecting across the table.** A row's ✓ or ✗ is per-element; the value is in the cross-element patterns (redundancy, disconnect). Pattern detection is Step 4 for a reason.

- **Audit without screenshots.** Like ux-audit, every interaction-audit references the captured artifact. "I read the code and concluded the button is dead" — show me the screenshot of the button + the captured handler trace.

- **Conflating interaction-audit with visual-audit.** They're orthogonal. The agent should refuse visual-polish gradings ("the spacing is wrong") and route to ux-audit. Conversely it accepts findings like "this card chrome reads as tappable but isn't."

- **Skipping ambiguous (⚠) findings.** Ambiguity is itself a finding — the user can't tell what the affordance does, which is a UX failure. Don't only flag clear ✗ cases.

## Tool surface

The agent needs: `Read`, `Grep`, `Glob`, `Bash`, plus the platform's capture + interaction tools (tap / type / inspect hierarchy).

Model: high-capability. The "what does this chrome promise" reasoning is non-trivial.
Effort: high. Per-element behavioral tracing across a non-trivial screen takes time.
