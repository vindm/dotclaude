# File-size rule (non-negotiable)

**Hard ceiling: 1000 LOC per source file.** No `.ts`, `.tsx`, or `.swift` file may exceed this. Decompose before merging — never as a follow-up ticket.

## Why

- 2000+ LOC files burn the context window on every edit and make the LLM pay tokens to re-read what's already there.
- Reviewers can't hold the whole file in their head; drift between sibling code paths goes unnoticed.
- Hooks-shaped state machines stop being legible past ~400 LOC; persistence pipelines past ~600 LOC.
- The cost of decomposing a 1500-LOC file is paid once; the cost of editing it is paid every change forever.

## How to decompose

| Pattern in the bloated file | Extract to |
|---|---|
| Stateful effect + refs + callbacks scoped to one feature | `useFeature.ts` hook |
| Pure transformation function + helpers | `src/.../operations/<verb>.ts` module |
| > 50 LOC of inline types | sibling `types.ts` |
| Per-action dispatch / handler bundle | `useFeatureHandlers.ts` returning `{ handle*, state }` |
| Sub-component > 100 LOC inside a screen | `<ChildComponent>` in its own file |

For React screens specifically: extract hooks first (state + effects + callbacks), then JSX sub-components, then pure helpers — in that order.

## Workflow

- After any edit batch, `wc -l` the touched files. If any is at 950+, decompose now while context is loaded.
- If a single feature genuinely needs > 1000 lines of cohesive code, that's a signal the feature is two features — split conceptually, not just textually.
- Don't game the rule with `// eslint-disable` files, `.barrel.ts` re-exports of one giant module, or splitting a function across files just to dodge the count.

## Exemptions

- Auto-generated files: `database.types.ts`, `*.gen.ts`, anything with `// AUTOGENERATED` header.
- Test fixture files holding raw recorded data.
- Snapshot files (`__snapshots__/`).

Everything else: 1000 LOC max. The refactor lands in the same PR as the work that pushed the file over — not a backlog item.
