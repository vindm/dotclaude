# decompose-file

You are decomposing a file that's at or over the 1000-LOC ceiling. The rule is in your project's `file-discipline` rule — read it first.

## Why this matters

The project's rule: "No file > 1000 LOC. Decompose in the same change. `wc -l` after edits." 2000+ LOC files burn the context window on every edit, hide drift between sibling code paths, and cost more to edit forever than to decompose once.

## Inputs

User passes a path. If they don't, ask. Verify the file is over the ceiling:
```bash
wc -l <path>
```

## Workflow

### Step 1 — Read and classify

Read the whole file. Identify the patterns from your `file-discipline` rule's extraction table:

| Pattern | Extract to |
|---|---|
| Stateful effect + refs + callbacks scoped to one feature | `useFeature.ts` hook |
| Pure transformation function + helpers | `src/.../operations/<verb>.ts` module |
| > 50 LOC of inline types | sibling `types.ts` |
| Per-action dispatch / handler bundle | `useFeatureHandlers.ts` returning `{ handle*, state }` |
| Sub-component > 100 LOC inside a screen | `<ChildComponent>` in its own file |

For React screens specifically: **hooks first, then JSX sub-components, then pure helpers**, in that order.

### Step 2 — Propose the split (don't edit yet)

Present a plan to the user:

```
File: <path> (<N> LOC)

Proposed split:
1. <hook 1> → <new path> (~<LOC> LOC)
2. <hook 2> → <new path> (~<LOC> LOC)
3. <subcomponent> → <new path> (~<LOC> LOC)
4. <pure helpers> → <new path> (~<LOC> LOC)

Resulting parent file: ~<LOC> LOC
```

Sanity checks before presenting:
- Each new file should land 100–500 LOC. If a chunk would be < 30 LOC, fold it back instead of creating a tiny file.
- The parent file should hold composition + JSX root + non-extractable wiring only.
- If the parent stays > 800 LOC after the split, the feature is two features — flag it conceptually and ask whether to split into sibling features.

### Step 3 — Get approval, then execute

Wait for the user to confirm or amend the plan. After approval:

1. Create new files one at a time.
2. After each extraction, **run `wc -l`** on the touched files.
3. After all extractions, run your lint + test commands on the touched files.
4. Verify the parent is under 1000 LOC; if not, iterate.

### Step 4 — Don't game the rule

Per your `file-discipline` rule:
- Don't `// eslint-disable` files to skirt the limit.
- Don't make `.barrel.ts` re-exports of one giant module.
- Don't split a function across files just to dodge the line count.

If you're tempted to do any of these, the right answer is **"this feature is two features"** — split conceptually.

## Exemptions (don't decompose these)

- Auto-generated files: `database.types.ts`, `*.gen.ts`, anything with `// AUTOGENERATED` header.
- Test fixture files holding raw recorded data.
- Snapshot files (`__snapshots__/`).

If the user invokes you on one of these, point it out and refuse politely.

## Don't

- Don't decompose without showing the plan first.
- Don't extract aggressively into 5-file fragmentation; aim for 2–4 sibling files.
- Don't skip the `wc -l` check after editing — proves the rule is satisfied.
- Don't split tests across multiple files unless the test suite was already structured that way.
